Honest scope (why a generic repo)
---------------------------------
The intent was a small public gitlab.com repo that actually contains LikeC4
files (.c4/.likec4/.likec4.snap) so the §8 file-type FILTER is exercised on the
keep side against real bytes. A genuine, exhaustive search of gitlab.com found
NONE that are reachable unauthenticated:
  - projects?search=  for likec4 / likec4-example / c4-model / c4 dsl /
    model.likec4 / landscape.likec4 / diagrams-as-code / architecture-as-code /
    structurizr / ...  (GitLab name-search does NOT index file contents)
  - projects?topic=   likec4 (empty), c4 (3 hits), c4-model, c4model
  - ~20 candidate repo trees probed recursively (tree?recursive=true) -> 0 .c4/.likec4
  - the LikeC4 author's gitlab.com account (davydkov) -> no public projects
  - group "likec4" -> 404
  - GitLab /search?scope=blobs (the only content search) -> 401, requires a
    logged-in token; none available at record time.
LikeC4's ecosystem lives on GitHub, not gitlab.com.

So this is the spec-sanctioned FALLBACK: a small, stable, generic public repo.
Against these REAL bytes the contract test (GitLabRealContractTest) asserts:
  - resolveSha parses the real commits JSON and extracts the pinned 40-hex id,
    robustly despite all the surrounding real fields;
  - the real archive.tar.gz extracts correctly: the real "<proj>-<sha>-<sha>/"
    top dir is stripped and repo-relative nested paths are exact (whole repo and
    a path= subtree), using the same gzip/tar + PathSafety as production;
  - the §8 LikeC4 filter, run end-to-end through GitLabSourceClient, correctly
    DROPS every one of the 50 real non-LikeC4 files (keep-set is empty for this
    repo) -- the drop side is mutation-checked non-vacuous (50 real files dropped);
  - the PRIVATE-TOKEN header and path= scoping travel on the real calls.
