# CLAUDE.md

Guidance for agents working in this repo. Keep it accurate — update it when build commands, the target version, or the gotchas change.

## What this is
A self-contained **Confluence Data Center 10.2.13** P2 plugin that renders fully-interactive **LikeC4** diagrams in the browser, sourcing `.c4`/`.likec4` DSL + `.likec4/*.likec4.snap` manual-layout snapshots from a **self-managed GitLab**. Target platform: **Jakarta EE** (`jakarta.*`, not `javax.*`), **Java 21**, likec4 **1.58.0** (pinned).

## Modules
- `likec4-core/` — Atlassian-independent Java (GitLab fetch, archive extract, file-type filter, caches, token crypto, allowlist, input validation). 255 JUnit tests. Pure Java; **no `jakarta`/Confluence APIs**.
- `src/main/frontend/` — Vite 5 / React 19 / TypeScript browser bundle (worker-pool compute via `@likec4/language-services`, `likec4/react` render, IndexedDB cache, the macro-editor "Load views" picker). 244 vitest tests + typecheck. Playwright e2e lives in `docker/e2e/`.
- `src/main/java/com/phrontizo/confluence/likec4/` + `src/main/resources/atlassian-plugin.xml` — the Confluence wrapper (macro, REST resources, admin servlet, admin config, web-resources). Jakarta EE. 127 JUnit authz/WebSudo tests (`src/test/java`, JUnit 5 + Mockito) run during the plugin build.
- `docker/` — compose harness (Confluence 10.2.13 + Postgres + mock-GitLab), unattended setup (`up.sh`), and the live/Playwright e2e. See `docker/README.md`.

## Build & test (verified commands)
Requires **JDK 21** (Confluence 10's platform — `JAVA_HOME` set or `java` on `PATH`), **Node 20**, and the committed **`./mvnw`** wrapper (Maven 3.9.9). Confluence 10.2.13 deps resolve from the public Atlassian repo (no creds).

- Core tests: `./mvnw -B -ntp -f likec4-core/pom.xml test`
- Frontend: `cd src/main/frontend && npm ci && npm run typecheck && npm test`
- Full plugin JAR (runs the 127 wrapper JUnit tests, compiles the Jakarta wrapper against Confluence 10.2.13 + builds the frontend bundle):
  ```
  ./mvnw -B -ntp -f likec4-core/pom.xml install -DskipTests   # install core FIRST (see gotcha)
  ./mvnw -B -ntp package                                      # runs wrapper tests -> target/likec4-confluence-*.jar
  ```
  (add `-DskipTests` to skip the wrapper unit tests for a faster build — note this skips only the Java unit tests; the frontend `npm ci`/`npm run build` is bound to `generate-resources` and still runs.)
- **Live e2e gate (REQUIRED — see below): `docker/up.sh` (once, to bring the stack up) then `docker/e2e/c10-gate.sh`** — (re)installs the freshly-built JAR into the running Confluence 10.2.13 and runs the 3 runtime gates (GATE3 macro render, GATE4 admin servlet — the config form loads for an authenticated admin and an unauthenticated caller is bounced; the harness auths via WebSudo-exempt Basic, so cookie-session WebSudo enforcement is proven separately, see below, GATE5 macro editor) via containerised Playwright. Exits non-zero on any failure. Needs Docker, a timebomb DC licence (`docker/.timebomb-license`, gitignored), and several GB of RAM.
- Other live/Docker e2e (full sweep, real-GitLab, perf): `docker/README.md`.

CI (`.github/workflows/ci.yml`) runs the first two + the plugin build (incl. the 127 wrapper unit tests) on every push/PR. The **live e2e gate is NOT run in CI** (it needs Docker + a licence + RAM) — it is the local/agent gate, see below.

## The code is NOT "working" until the live e2e gate passes
Green unit tests (core + frontend + wrapper) prove the pieces; they do **not** prove the plugin renders, the admin/WebSudo flow works, or the editor authors a macro on a real Confluence. Those are only proven by the **live e2e gate** (`docker/e2e/c10-gate.sh` → GATE3/4/5 all pass). Treat a change as done/working **only** when both are green:
1. unit: `255 core + 244 frontend + 127 wrapper + typecheck` and the full `./mvnw -B -ntp package`, AND
2. live: `docker/e2e/c10-gate.sh` exits 0 with all 3 gates passing.
The compose stack is long-lived; bring it up once with `docker/up.sh`, then re-run `c10-gate.sh` after each change (it reinstalls the current JAR each run). **`c10-gate.sh` only redeploys the plugin JAR — it does NOT rebuild the mock-GitLab image.** If you change anything under `docker/mock-gitlab/` (e.g. `server.mjs`), re-run `docker/up.sh` (which does `up -d --build`) or `docker compose -f docker/compose.yaml up -d --build mockgitlab` first, or the gate silently tests against the stale `likec4-mock-gitlab:dev` image (see the comment in `docker/compose.yaml`).

## Critical gotchas (do not relearn these the hard way)
- **NEVER run `mvn clean` / `./mvnw clean`** while a local Confluence is running — it wipes `target/confluence/home` and kills the instance. To refresh the web-resource bundle: `rm -rf target/classes/likec4-web && ./mvnw -o package`.
- **The root plugin build depends on `likec4-source-core` being installed to `~/.m2`** (it's a standalone module, not a reactor child). Install it after changing `likec4-core/`, or the wrapper won't find it. **After changing `likec4-core` on a dirty tree, force a clean re-bundle of core before the wrapper build:**
  ```
  rm -rf target/classes/com/phrontizo/likec4 target/dependency-maven-plugin-markers target/fe-bundled-metadata-markers
  ```
  Core is the sole compile-scope (bundled) dep, so amps unpacks its classes INTO the plugin's `target/classes` at `package`, guarded by marker files under `target/dependency-maven-plugin-markers` / `target/fe-bundled-metadata-markers`. Two DISTINCT dirty-tree failures result, and clearing only ONE half makes the OTHER worse:
  - *Compile shadow:* a stale unpacked `SourceService.class` (etc.) sits FIRST on the compile classpath and shadows the freshly-installed dependency jar → the wrapper fails to compile with a phantom `cannot find symbol` against the very method you just added.
  - *Bundle skip:* the marker persists, so amps reports `already exists in destination` and SKIPS re-unpacking core. If you cleared `target/classes` (or just `…/com/phrontizo/likec4`) but LEFT the markers, the built JAR ships with ZERO core classes → the plugin fails to enable at runtime with `NoClassDefFoundError: com/phrontizo/likec4/source/…` (e.g. `TokenCipher$DecryptException`), an opaque "plugin did not enable in time" in the e2e gate. **Verify the fix:** `jar tf target/likec4-confluence-*.jar | grep -c com/phrontizo/likec4/` must be non-zero.
  CI never hits either (it always builds clean); local/loop incremental builds hit both. **Do NOT `rm -rf target/classes/com/phrontizo/likec4` alone** — clear the markers too.
- **Edit the hand-written classic scripts (`boot-loader.js`, `editor-loader.js`) in `src/main/frontend/public/`, NOT the `src/main/resources/likec4-web/` copy.** Vite (`outDir: ../resources/likec4-web`, `emptyOutDir: true`) WIPES and regenerates `src/main/resources/likec4-web/` from `public/` + the bundle on every `package`/`npm run build` — so edits to the `resources/likec4-web/` copy are silently overwritten and never reach the JAR. (`src/main/resources/likec4-web/` is a generated, gitignored artifact — not committed.)
- **Target is Jakarta EE** — use `jakarta.ws.rs` / `jakarta.servlet` / `jakarta.inject` (NOT `javax.*` for EE APIs). The port from Confluence 9.2 was primarily this namespace swap (the compile target also moved to Java 21 and jackson was realigned to the platform-managed version). (JDK packages such as `javax.crypto` in `TokenCipher` are *not* EE namespaces and are unaffected — leave them.)
- **REST/servlet DI:** JAX-RS resources are HK2-instantiated and can't inject Spring/SAL beans → wire via `atlassian-plugin.xml` `<component>`/`<component-import>` + a static `INSTANCE` set in the bean's ctor; resources read `getInstance()`. The macro uses spring-scanner (`@ConfluenceComponent` + `@ComponentImport`).
- **Browser load path:** Confluence batches web-resources, so `boot-loader.js` reconstructs the absolute `/download/resources/<module-key>/likec4-web/assets/*` URLs (entry JS **and** the entry CSS sidecars from the Vite manifest) rather than deriving from `document.currentScript.src`. REST base uses `window.AJS.contextPath()`.
- **Reinstalling the plugin into a running Confluence: UNINSTALL + INSTALL, don't "upgrade".** A UPM upgrade of the *same* `0.1.0-SNAPSHOT` version does NOT reliably redeploy new bytecode (OSGi caches it) — the live behaviour silently stays on the old build, which will make e2e results lie. `docker/e2e/c10-gate.sh` does a clean uninstall+install (and re-configures admin, since uninstall wipes plugin settings) for this reason.
- **`commons-compress` MUST be bundled into the plugin — and a warm bundle cache HIDES it if it isn't.** Confluence 10.2.13 ships `commons-compress-1.28.0.jar` in `WEB-INF/lib` (that's why the version is pinned to match) but does **not** OSGi-export that package to P2 plugins, so amps treats it as product-provided and leaves it out of the bundle — making the archive extractor's `GzipCompressorInputStream`/`TarArchiveInputStream` a runtime `NoClassDefFoundError` on every cache-**miss** fetch (jackson works only because Confluence *does* export it). The root `pom.xml` unpacks commons-compress into the plugin classes at `prepare-package` (`maven-dependency-plugin` → `unpack`); **do not remove that** or `/source` breaks live. This shipped broken and hid for a long time because the on-disk bundle cache (`${java.io.tmpdir}/likec4-bundle-cache-<user>`, **not** wiped on uninstall) served the demo paths without ever extracting — a green GATE3 on a warm cache does NOT prove extraction works. `c10-gate.sh` therefore **clears that cache before the gates**; to check extraction by hand, fetch a **fresh ref** (the mock derives the sha from `project@ref`, so a novel ref forces a real extraction).
- **`BaseUrlValidator` intentionally allows plain `http` for internal/private hosts** (loopback, RFC1918, single-label intranet names, internal TLDs) — a self-managed GitLab is frequently internal http, and the harness uses `http://mockgitlab:8080`. `https` is required only for *public* FQDNs. Don't "harden" it to https-only: it compiles + unit-passes but breaks live setup (GATE3) and real internal deployments (caught only by the e2e gate).
- **WebSudo vs the Basic-auth harness:** the admin *page* (`AdminServlet`) enforces a Secure Administrator Session for interactive (cookie-session) callers but **exempts HTTP Basic auth** (a Basic-auth request already presents the password each call, so WebSudo's re-auth is redundant, and the test stack is Basic-auth-only — form login is disabled on it). Don't "fix" GATE4 by removing the exemption; the cookie-session enforcement path is verified separately (an un-elevated session 302s to `/authenticate.action`).
- **Don't chase the macro-browser `reading 'macroName'`/`reading 'body'` console error** — it's a *known-benign upstream Confluence bug*. Its macro-browser PREVIEW (`AJS.MacroBrowser.previewOnload` + the preview iframe's own `about:srcdoc` onload) null-derefs `dialog.activeMetadata.macroName`/`.body` for ANY macro using a `setMacroJsOverride` opener (ours AND Confluence's own `jira*` macros — the override bypasses the default dialog that sets `activeMetadata`). It's in uneditable platform code (priming `activeMetadata` only swaps `macroName`→`body`; srcdoc-setter/setAttribute/MutationObserver interception can't reach the platform-parsed iframe). Authoring is unaffected; GATE5 filters it via `isBenignMacroBrowserPreview` (a `reading '(macroName|body)'` matcher that is stack-scoped — it ignores the error only when the stack does NOT reference `likec4-web`, so a genuine regression from our own bundle still fails the gate; `IGNORED_PAGEERROR` is the *separate* emoji/ResizeObserver chrome-noise filter).
- **Manual-layout snapshots** (`.likec4/*.likec4.snap`) must be internally coherent (node positions, edge geometry, and `view.bounds` consistent) — `applyManualLayout` renders snapshot geometry verbatim. `compute.ts` re-derives `view.bounds` after applying. (The filter requires the full `.likec4.snap` suffix; a bare `.snap` is dropped — see `LikeC4FileFilter`.)
- **npm audit is not a CI gate, and you cannot cleanly clear it via `npm audit fix --force`** — that downgrades `likec4` below the hard-pinned 1.58.0. The frontend pins an `overrides` entry forcing `lodash-es` to 4.18.x (clears the high-severity advisory that came in transitively via langium/chevrotain). The residual `npm audit --omit=dev` lows are `esbuild` *dev-server* build tooling bundled inside the `@likec4/*` packages — not reachable by the shipped browser bundle. Don't "fix" them by downgrading likec4.
- **The `npm ci` `EBADENGINE` warning for `likec4`/`@likec4/language-server` (wants Node `>=22.22.3`, we run Node 20) is EXPECTED — do not "fix" it.** (`@likec4/language-server` is a *transitive* package one letter apart from our *direct* dep `@likec4/language-services` — the latter declares no engine floor; the warning comes from the former and from `likec4` itself.) likec4 1.58.0 raised its Node floor above Confluence 10's Node-20 platform, but that floor is for its *CLI*; the plugin ships only likec4's *browser* code (which runs fine on Node 20 — 244 vitest + the live e2e gate prove it). Bumping the pinned Node breaks the Confluence-10/Java-21/Node-20 alignment, and downgrading likec4 is forbidden (see above) — so the warning stays.

## Conventions
- Verify before claiming done: 255 core + 244 frontend + 127 wrapper + typecheck green, **AND the live e2e gate `docker/e2e/c10-gate.sh` green** (GATE3/4/5). For UI/render changes, *look at* the result (the `docker/e2e/*.spec.ts` guards assert real visibility/coherence, not just DOM presence). Unit-green alone is NOT "working".
- Keep `docker/.timebomb-license` and any real tokens out of git (gitignored). Dev-only JVM flags (Basic-auth force-allow, UPM signature-check disable) are in `compose.yaml` for the test harness only — production uses PAT/OAuth + a signed build.
